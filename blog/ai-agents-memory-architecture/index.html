My AI Agents Nearly Drowned in Their Own Memory. Here's What I Rebuilt.

Rajan Rengasamy · March 7, 2026 · 5 min read

The model never failed. The infrastructure around it did.

Last week my content agent's session hit 160,000 tokens out of a 200,000 limit. Twenty compaction attempts failed. Fourteen timeouts in nine days. The agent wasn't broken. It was suffocating under 82,000 characters of context it loaded before saying a single word.

One agent's working memory file had ballooned to 50KB. Cron jobs were silently appending to daily logs using find-and-replace that failed on stale content. Nobody noticed because the agents kept running. They just got worse.

That's the failure mode nobody warns you about with AI agents. The model doesn't degrade. The memory does.

What I'm Running

Five persistent agents on three Apple Silicon machines. A CTO agent, a scheduler, a content CMO running Opus 4.6, a creative agent, and a builder. 54 cron jobs. 211 overnight builds. They coordinate through a shared SQLite database, JSON state files, and Discord channels.

Most of it runs invisible. Then there's the agent bus.

They post status updates, build results, pipeline completions, and escalation requests to each other. The first time I scrolled through it after a night of builds, I found messages timestamped 2am: three agents completing a handoff between them, eleven minutes, no human in the loop. It felt like reading a shift log from a team I'd never met.

That infrastructure worked. The memory system underneath it didn't.

What Broke

My original design was naive in a specific way: load everything. Every session, every agent got its full identity, rules, tools, memory, and daily logs.

I thought more context meant smarter agents. It doesn't. It just means more noise competing with fewer signals.

The content CMO's system prompt alone was 82,000 characters. Daily notes hit 35,000 characters. Working context files that were supposed to be scratchpads became append-only logs.

When the system tried to compact the session to free space, the context was so bloated that compaction itself timed out. The agents kept functioning. They just lost precision.

Instructions from three weeks ago competed with instructions from today. Stale context diluted fresh context. The agent "knew" everything and prioritized nothing.

The Architecture That Fixed It

Three tiers. Hard caps. No exceptions.

Tier 1: Identity. Six files, 30 to 50KB total. Capped at 10KB each. Before I enforced caps, these were running 80 to 110KB of accumulated drift. That's not an identity file. That's a graveyard.

Tier 2: Working state. A scratchpad at 5KB, overwritten on state changes, never appended. A memory index at 8KB that tells the agent what to search for, not what to remember. Last two days of compressed daily notes, 5KB each. Fixed window. Nothing accumulates.

Tier 3: Deep storage. Two backends, both local. LanceDB for agent memories: every fact vector-embedded and stored for semantic search, recalled by meaning not keywords. SQLite for workspace files: full-text search across indexed file chunks. Neither loads at startup. Both queried on demand. No cloud, no context cost.

The principle: an agent with infinite context has no focus. Recency and relevance govern what loads. Everything else gets searched.

The caps are the architecture. When a file hits its limit, old content gets archived, not accumulated. This is the part most people skip. They build the memory system and forget to build the forgetting system.

What Changed

Session sizes dropped by half. Compaction failures stopped. The agents didn't get new capabilities. They got focus.

That was the fix. Then I found something in the overnight logs I hadn't planned for.

28 out of 170 overnight builds independently included safety components that weren't in the spec. Rate limiters. Output validators. Dry-run modes. Human approval gates.

The context files shaped that behavior. When agents load memory that includes past failures, operational constraints, and review standards, they build more conservatively. Not because they're reasoning about safety. Because the context biases them toward patterns they've already seen.

Context is culture. Shape the files, shape what gets built.

The Actual Point

Everyone's debating which model is best. I've swapped models three times this month. The agents barely noticed. What they notice immediately is when their context is wrong, stale, or bloated.

The model is the easiest part to replace. The memory architecture, the coordination layer, the review gates, the forgetting system: that's the infrastructure that determines whether your agents actually work in production or just demo well.

I built this between 10pm and 2am in Sydney. It's not finished. But every morning I wake up to a shift log from a team that worked through the night, and the system is better than it was yesterday.

The model is replaceable. The system is the product.
