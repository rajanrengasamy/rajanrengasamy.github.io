Nate Built a Brain. I Built a Nervous System. We Solved the Same Problem Differently.

March 15, 2026 · @cmd_alt_ecs agents memory architecture

Nate B. Jones just released 6 extensions for his Open Brain system. I've been running 5 AI agents with shared memory across 3 machines for months. We arrived at the same insight from opposite directions.

The Problem We Both Noticed

Every AI tool forgets you the moment you close the tab. Open a new Claude chat, a new ChatGPT thread, a new Cursor session. Each one starts from zero. Your preferences, your projects, your decisions, your relationships. Gone.

Nate calls this the "junk drawer problem." I call it the cold start tax. Different names, same frustration: your AI should know you by now.

Nate's Answer: One Brain, Many Clients

Open Brain is elegant in its simplicity. One Supabase database. One Slack channel for capture. One MCP server that any AI client can connect to. Type a thought into Slack, and an edge function classifies it, extracts metadata, embeds it, and stores it. Claude, ChatGPT, Cursor, whatever ships next month. They all read from the same brain.

His new extensions push it further. Six modules that build on each other:

  • Household knowledge base (your agent knows your WiFi password)
  • Home maintenance tracker (your agent reminds you about the furnace filter)
  • Family calendar (your agent checks who's home before planning meals)
  • Meal planning with shared grocery lists
  • Professional CRM wired into your captured thoughts
  • Job hunt pipeline with interview tracking

The key design principle: extensions compound. The CRM knows about people you've mentioned in captured thoughts. The meal planner checks who's home from the family calendar. Job contacts become CRM contacts automatically. One memory layer, multiple views on top.

Setup takes 45 minutes. No code. Costs about $0.10/month in API calls. Nate designed this for knowledge workers who've never opened a terminal.

My Answer: Five Agents, Shared State, SQLite

I went the other direction entirely.

My setup: 5 AI agents (Jarvis, Friday, Mercury, Vision, Edith) running across 3 machines (a MacBook Pro, a Mac Studio, a headless Mac Mini). 60+ scheduled crons. They coordinate through a shared SQLite database and file-based state at /Users/rajan/state/.

There's no single "capture" action. My agents generate their own context. Mercury scans 70+ signals per day across X, Reddit, Hacker News, YouTube, and GitHub. A triage pipeline scores them. A content producer drafts posts. An enhancer rewrites them in my voice. An editorial gate (also Mercury) kills anything that reads like AI slop.

Memory isn't one database. It's a 4-tier architecture:

TierWhatRead Frequency
Working MemoryPipeline state, pending items, blockersEvery session
Long-term MemoryStrategic learnings, voice rules, operating directivesEvery session
Daily LogsEditorial decisions, pipeline interventionsLast 2 days only
Cold StorageHistorical drafts, analytics, trend dataOn demand (vector search)

Each agent reads working memory and long-term memory on boot. Daily logs get archived after 7 days. Strategic learnings get distilled upward. It's less "brain" and more "institutional memory distributed across a team."

Same Philosophy, Different Tradeoffs

When I look at what Nate built and what I built, the core insight is identical: your AI context should outlive any single session, model, or tool.

But the tradeoffs are real:

Open BrainAgent Fleet
Setup45 minutes, no codeWeeks of iteration, deep config
Cost~$0.10/month$0/month (subscription models)
Memory modelSingle Supabase + vector search4-tier (working/long/daily/cold)
CaptureHuman types into SlackAgents generate autonomously
UsersKnowledge workers, non-technicalBuilders, developers, tinkerers
ExtensibilityAdd Supabase tables + MCP toolsAdd agents, crons, scripts
What breaksSlack webhook limits, classification errorsAgent coordination, memory overflow, dedup

Nate optimized for adoption. Minimum friction, maximum reach. His system works for a VP of product who captures thoughts on a commute.

I optimized for autonomy. My agents don't wait for me to type. They scan, triage, draft, review, and publish while I sleep. But the setup cost is measured in weeks, not minutes.

What I'd Steal From Open Brain

Nate's MCP server approach is smart. One URL, any client connects. My agents coordinate through files and database rows, which works but creates coupling. If I rebuilt the inter-agent communication layer, I'd seriously consider an MCP-based protocol where agents discover each other's capabilities through a shared registry rather than hardcoded paths.

His "extensions compound" principle is also something I've arrived at independently but never articulated as cleanly. My content scanner knows about my publishing calendar. My publisher knows about editorial decisions. My analytics collector feeds back into content strategy. It compounds. But I never designed it that way from the start. It grew organically, which means the connections are messier than they need to be.

What Open Brain Could Learn From Agent Fleets

Open Brain is still fundamentally capture-and-retrieve. You put things in, you pull things out. The AI is a librarian.

The step Nate hasn't taken yet is autonomous action. What if the system noticed you haven't contacted a CRM contact in 3 months and drafted a check-in? What if it saw a calendar conflict and proposed a resolution before you opened Slack? What if it watched your job hunt pipeline and proactively researched companies when applications went cold?

That's where agent architectures start to pay off. Not just memory, but judgment. Not just storage, but initiative. The extensions point in this direction. The compounding is already there. The missing piece is agents that act on what they know without being asked.

The Real Lesson

Both approaches share a conviction that feels obvious but almost nobody implements: AI context is infrastructure, not a feature.

It's not about which model you use. It's not about prompt engineering. It's about building a system where your AI remembers who you are, what you've done, what you're working on, and what you care about. Across every tool. Across every session. Permanently.

Nate built a clean, accessible version of this for everyone. I built a messy, powerful version for one person (me). Both work. Both compound. The best choice depends on whether you want to capture your thinking or automate your workflow.

Or maybe, eventually, both.
