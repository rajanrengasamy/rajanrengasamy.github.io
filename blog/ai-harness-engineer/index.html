I Think I Accidentally Became an AI Harness Engineer.

Rajan Rengasamy · March 15, 2026 · 5 min read

I didn't know the job had a name until OpenAI published a blog post describing exactly what I do every night from 9pm to midnight.

OpenAI shipped a million-line beta product with zero manually-written source code. A small team of engineers guided Codex agents through pull requests and CI workflows. They provided prompts and feedback. The agents iterated autonomously on tasks including reproducing bugs, proposing fixes, and validating outcomes.

They called the methodology "harness engineering." Martin Fowler endorsed it on LinkedIn. Philipp Schmid wrote the definitive technical breakdown. Aakash Gupta declared "2025 was agents. 2026 is agent harnesses."

I read the OpenAI blog post and realized they were describing my Tuesday nights.

The Setup Nobody Asked For

I run five AI agents across three machines. A CTO agent that builds overnight. A CMO agent that publishes content. A COO agent that manages my calendar. An editorial agent that reviews everything before it goes live. A builder agent that ships code to GitHub while I sleep.

Each has its own system prompt, its own memory architecture, its own tool access rules, its own model routing. The CTO gets Codex for greenfield builds. The CMO gets Opus for editorial judgment. Quick lookups get Haiku. 60+ automated crons coordinate the whole thing.

I didn't plan to become a harness engineer. I just needed my agents to stop breaking each other.

The Six Components

Aakash Gupta identified six components that make agent harnesses work. I checked my setup against all six. I'm running every single one.

I built all of this before I knew there was a name for it. The problems forced the architecture.

The Problems That Forced the Architecture

Three weeks in, my content agent's system prompt hit 82,000 characters. Twenty compaction failures. Fourteen session timeouts in nine days. The agent was running but losing precision because its context window was packed with stale notes from conversations that ended weeks ago.

The fix wasn't a better model. It was a memory architecture with hard caps: 5KB working memory (overwritten, never appended), 8KB long-term memory (curated weekly), daily logs with 48-hour expiry. The agents didn't get new capabilities. They got focus.

Separately, cron jobs were silently corrupting shared files with stale find-and-replace operations. The failure was invisible until agents started behaving strangely. The fix: ban background processes from writing to shared state files. Read-only for crons, write access only for live sessions.

These aren't AI problems. They're systems engineering problems. Context management, state persistence, failure recovery. The same problems we solved in distributed systems a decade ago, repackaged for agents.

The Job Description Nobody Wrote

Harness engineering shifts your focus from implementing code to designing environments. You're not writing the logic. You're designing the world the agent perceives.

My actual job description, if I had to write one:

  • Define what each agent can and can't touch (filesystem boundaries)
  • Design how they hand work to each other (sub-agent coordination via state files)
  • Build the memory system so they wake up knowing what happened yesterday (lifecycle hooks)
  • Set up 60+ automated jobs with different model routing per task (tool orchestration)
  • Review output and feed corrections back into the system (human-in-the-loop)
  • Debug why an agent posted something embarrassing at 3am and make sure it never happens again (guardrails)

That last one isn't in any framework diagram. But it's half the job.

The Part Nobody Talks About

Philipp Schmid made an observation that matches my experience exactly: "To survive the Bitter Lesson, your infrastructure must be lightweight. Every new model release has a different, optimal way to structure agents. If you over-engineer the control flow, the next model update will break your system."

I've rebuilt my memory architecture twice. The cron system three times. The sub-agent coordination layer was completely rewritten when the daemon replaced the batch pipeline. Capabilities that required complex hand-coded pipelines three months ago are now handled by a single context-window prompt.

The skill isn't building the harness. It's building a harness you can tear apart and rebuild when the next model drops.

If You're Already Doing This

If you're running AI agents beyond a single chat session, you're probably doing harness engineering too. You just don't have the job title yet.

The term is useful because it names what practitioners are actually doing. Not prompt engineering (that's one small piece). Not vibe coding (that's the opposite of what this requires). Not "AI strategy" (that's a PowerPoint, not an architecture).

Harness engineering. The infrastructure layer between the model and the outcome.

I didn't choose this job. The agents broke enough things that the job chose me.
