Same Model. 78% vs 42%. The Harness Made the Difference.

Rajan Rengasamy · March 12, 2026 · 4 min read

Claude Code gave it filesystem access and persistent memory. Codex sandboxed it and slid results under the door. 36-point gap from everything except the model itself.

Nate B Jones ran the same benchmark on the same model under two different harnesses. The results were 78% vs 42%. A 36-point gap. Not from a different model, not from a better prompt, not from fine-tuning. From the execution environment around an identical model.

HarnessScoreWhat it gave the model
Claude Code78%Filesystem access, persistent memory, full context

Same model. Same task. One had everything it needed in reach. The other had to work through a slot in the wall.

We Hit the Same Wall

I run five agents across three machines. They handle content, research, overnight builds, scheduling, and escalations. The model routing was the easy part: Opus for editorial judgment, Codex for greenfield builds, Haiku for quick lookups. That configuration took an afternoon.

Getting the memory architecture right took three weeks. And it nearly killed everything first.

Our MEMORY.md ballooned from 5KB to 50KB. The system prompt hit 82,000 characters. Twenty compaction failures. Agents started zombifying — running but losing precision — because their context windows were packed with stale notes from conversations that ended weeks ago.

The model was fine the whole time. The harness around it was drowning.

That's exactly what Nate's benchmark measured. Not model capability. Harness quality.

What We Rebuilt

Three tiers. Hard caps. A forgetting system.

Working memory — 5KB cap, overwritten on state changes, never appended. What the agent needs right now, nothing else.

Long-term memory — 8KB cap, curated weekly. Strategic learnings, permanent directives, things that need to survive across months. Everything else gets archived.

Daily logs — 48-hour expiry. Decisions, signals, interventions. After two days, they go cold. The agent doesn't carry last week's problems into today.

One rule we had to add that nobody talks about: ban background processes from using find-and-replace on shared files. Cron jobs were silently corrupting memory files with stale writes. The failure was invisible until the agent started behaving strangely. The fix was structural — read-only access for background processes, write access only for the live session.

The Five Lock-In Dimensions Nobody Measures

Nate identified five dimensions where harness choice creates lock-in that teams typically don't account for when they pick a tool:

  1. Memory persistence — does the harness maintain state between sessions or start cold every time?
  2. Tool access scope — what can the model actually reach? Files, APIs, system calls, nothing?
  3. Context architecture — what gets loaded at startup vs. queried on demand?
  4. Execution sandboxing — how constrained is the execution environment? What can't it do?
  5. Coordination model — can agents communicate, or does everything route through a human?

Most teams measure the model. Almost nobody measures these. But this is what produced the 36-point gap.

The Part That Should Worry Engineering Leaders

Once you've built your workflows around a harness, you're not switching. The switching cost isn't the model — models are interchangeable. The switching cost is the memory architecture, the tool integrations, the coordination layer, the context design. That's six to eighteen months of institutional knowledge.

The five decisions that lock you in are probably being made right now by whoever set up the team's first Cursor subscription, picked the default Claude Code settings, or chose which agent framework to use for the first proof of concept.

Teams that understand this choose their harness the way they choose their cloud provider — as an infrastructure decision, not a tooling decision.

What We Do Now

Session sizes dropped by half after the rebuild. Compaction failures: zero. The agents didn't get new capabilities. They got focus.

When you fix the harness, you don't need a better model. You need the same model to stop fighting its own context.

The model is the easiest variable to change. The harness is what actually determines the output.
