Executive Summary

Program Manager Skill is an open-source AI agent skill that transforms Claude Code and Cursor into program management powerhouses. It automates the drudgery—meeting transcription, decision tracking, capacity planning, status reports—freeing PMs to do strategic work.

The Insight: PMs already use AI coding tools. Why not teach those agents to do program management?

Not a SaaS product. Not a chatbot. A skill.

Target: 50,000+ program managers using Claude Code or Cursor who spend 40% of their time on manual coordination work.

Business Model: Open-source core → GitHub traction → LinkedIn visibility → consulting inbound.

Success Metrics (6 months):

  • 1,000+ GitHub stars
  • 10 LinkedIn posts, 50k+ impressions
  • Internal Optiver PMO adoption
  • 3+ consulting inquiries

Problem Statement: The Real Pain

Context: Optiver's APAC PMO meeting (March 10, 2026) surfaced universal PM pain points. These quotes are from the transcript—real problems, not assumptions.

Pain Point #1: Meeting → Action → Follow-up is Manual Hell

"I record Zoom meetings → summaries → tickets → follow-ups, links meeting notes to technical debt backlog (400 days identified)"

Reality: PMs spend 40% of their time transcribing, creating action items, chasing follow-ups. High-value coordination done with low-value manual labor.

Pain Point #2: No Decision Log or Audit Trail

"We would like to formalize that because that ideally would help us to go and get additional resources if there is a lot of declined PM or DM requests"

Reality: Decisions made in meetings vanish into the void. Six months later: "Why did Project X get a PM and Project Y didn't?" No data. Just guesses.

Pain Point #3: Capacity Planning is Spreadsheet Archaeology

"Current state: Field-based insights vs. data-driven decisions"

Reality: Who's overallocated? Which projects are starving for resources? "Can we take on Project Z?" answered with "I think so?" instead of data.

Pain Point #4: Technical Debt is Invisible Until It Bites

"400 days of technical debt identified"

Reality: Debt lives in Jira backlogs, disconnected from strategy discussions. PMs can't answer "How much debt?" or "What's the cost?" without manual archaeology.

Pain Point #5: Stakeholder Communication = Repetitive Drudgery

Reality: Weekly status reports, steering decks, RAID logs—same format, different data. Hours spent reformatting information that already exists in Jira.

Pain Point #6: PMO is Execution-Focused, Not Strategic

"We are more on the execution side of things. Rather than the strategy, but that's where we'd like to move"

Reality: Too busy coordinating to answer strategic questions: "Should we invest in X or Y?" "What's our portfolio health?"

Pain Point #7: Tools Move Faster Than Adoption

"Tools move so quickly that you kind of go, all right, you've spent so much time figuring out how to do this... And then you go on leave for a few weeks, come back, and then all these new tools"

Reality: Tool proliferation. Each requires learning, setup, integration. No single "PM skill" that works everywhere.

The Opportunity

Observation: PMs at Optiver (and everywhere) already use AI coding tools—Cursor for dashboards, Claude Code for automation.

Insight: What if those same agents could do PM work natively? Not as a separate tool. As a skill.

Market Gap: Thousands of PM SaaS tools exist. ZERO skills that teach AI agents to be expert program managers.

Core Features (MVP vs. Post-MVP)

✅ MVP (Build Now - Demo-Ready in 2 Weeks)

Feature 1: Meeting Intelligence Pipeline ⭐ CORE DEMO

User Story: Process meeting transcripts → actionable work items automatically.

Capabilities:

  • Ingest Zoom/Teams/Granola transcripts
  • Extract action items, decisions, owners, deadlines
  • Auto-create Jira tickets via API
  • Save meeting summary to markdown

Example:


User: Process last week's PMO meeting transcript
Agent: Found 8 action items, 3 decisions, 2 risks
Agent: Created 5 Jira tickets (APAC-123 through APAC-127)
Agent: Summary saved to memory/meetings/2026-03-10-pmo.md

Feature 2: Decision Log & Tracking ⭐ MVP

User Story: Queryable log of all program decisions with rationale.

Capabilities:

  • Capture decisions (manual entry or extracted from meetings)
  • Store with metadata: date, participants, rationale, options considered
  • Natural language search: "Why did we decline a PM for Project Zebra?"

Example:


User: Why didn't we assign a PM to M365 migration?
Agent: Decision ABC-2024-Q3-014: "M365 Migration - PM Request Declined"
       Date: Sept 15, 2024
       Rationale: Team estimated <3 months, no cross-regional dependencies
       Outcome: Took 5 months, missed compliance dependencies
       Lesson: Assign PM to any compliance-touching project

Feature 7: Stakeholder Communication ⭐ MVP

User Story: Auto-generate status reports from Jira + meeting notes.

Capabilities:

  • Pull Jira data (tickets, velocity, milestones)
  • Pull recent decisions and meeting notes
  • Generate markdown status report from template
  • Output to Slack, email, or file

Example:


User: Generate weekly status report for TMT
Agent: Generated "2026-03-11-tmt-update.md"
       - Project status (3 on-track, 1 at-risk)
       - Risks (vendor delay, capacity constraint)
       - Decisions (1 pending: India Tax PM assignment)
       - Next actions (4 items)
Agent: Post to Slack #pmo-updates?

🟡 Phase 2 (Important, Not Blocking Demo)

Feature 3: Capacity & Resource Planning

User Story: Real-time visibility into team capacity and allocation conflicts.

Capabilities (Lite):

  • Manual capacity entry (person, project, % allocation)
  • Overallocation detection
  • Output markdown table

Deferred to Phase 2:

  • Integration with Jira time tracking
  • Gantt charts / visualization
  • Calendar integration for OOO

Feature 4: Intake & Request Formalization

User Story: Standardized project intake with priority scoring.

Capabilities:

  • Structured intake form (YAML/JSON)
  • Priority scoring based on business value, urgency, complexity
  • Track declined requests with rationale

Why Phase 2: Useful for headcount justification, but not demo-critical.

Feature 6: Technical Debt Quantification

User Story: Link technical debt to strategy discussions, calculate cost.

Capabilities:

  • Tag Jira tickets as debt
  • Calculate total debt (story points → days → dollars)
  • Basic cost model

Deferred to Phase 2:

  • Auto-extraction from meeting notes
  • Trend analysis
  • Prioritization engine

⚪ Nice-to-Have (Future)

Feature 5: Cross-Regional Coordination

User Story: Timezone-aware coordination, duplicate detection across regions.

Why Nice-to-Have: Niche use case, complex implementation. Defer until core features proven.

Feature 8: Strategic Advisory Engine

User Story: Portfolio-level analysis, investment prioritization.

Why Nice-to-Have: Requires full portfolio data pipeline. Impressive but not essential for MVP.

Key Design Decisions

1. Local-first, Git-friendly: Enterprise data stays on-premise. Works offline.

2. MCP optional, not required: Works with direct API calls if MCP unavailable.

3. LLM-agnostic: Works with Claude, GPT, Gemini—whatever user has.

4. Helper scripts for heavy lifting: Parsing Jira API better done in Python/Node than pure prompting.

5. Privacy-aware: No telemetry, no cloud dependencies (except user-configured APIs).

Integration Strategy

Jira Integration ✅ MVP (Works Today)

Approach: Direct API first, MCP optional.

Why: Jira is the system of record. Must work reliably.

Implementation:

1. Direct API (fallback, always works):

  • User provides: Jira URL, API token
  • Scripts use Jira REST API (create tickets, search, update)
  • Stored in ~/.pm-skill/config.json

    • 2. MCP Server (optional, cleaner):

  • If user has Jira MCP server configured, use that
  • Provides: jira.search, jira.create_ticket, jira.update_ticket
  • Agent auto-detects MCP availability, falls back to direct API if missing

Capabilities:

  • Search tickets (JQL queries)
  • Create tickets (project, summary, description, assignee)
  • Update tickets (status, fields, comments)

Confluence Integration ✅ MVP (Works Today)

Approach: Direct API first, MCP optional (same as Jira).

Capabilities:

  • Search pages
  • Get page content
  • Create/update pages
  • Publish meeting notes to Confluence space

Slack Integration ⚠️ API Only (MCP Not Ready)

Current Reality: Slack MCP via Cursor is NOT enabled as of March 2026.

Approach: Slack App (Bot Token) + Web API.

Setup Steps (documented in slack-app-setup.md):

1. Create Slack App at api.slack.com/apps

2. Add Bot Token Scopes: chat:write, channels:history, search:read

3. Install app to workspace

4. Copy Bot Token to ~/.pm-skill/config.json

Capabilities:

  • Search messages: POST /search.messages
  • Send messages: POST /chat.postMessage
  • Get channel history: GET /conversations.history

Implementation:

  • Use @slack/web-api npm package or direct REST
  • Scripts handle auth, rate limits, error handling

Future: When Cursor/Claude Code enable Slack MCP, add MCP path as primary with API fallback.

Calendar Integration ⚪ Nice-to-Have (Future)

Why deferred: Capacity planning works without it for MVP. OAuth flows add complexity.

If built later: Google Calendar API + Microsoft Graph API.

Databricks Integration ⚪ Nice-to-Have (Enterprise)

Competitive Landscape (Trimmed)

Direct Competitors (AI PM Tools)

Motion / Reclaim AI: Calendar-focused, doesn't handle meetings → actions.

Fireflies.ai: Stops at transcription, doesn't create tickets.

Spinach.io: Agile-focused, SaaS ($10-20/user), no portfolio features.

Differentiation: PM Skill is open-source, cross-platform, portfolio-level, designed for PMOs not solo users.

Indirect Competitors

Jira / Monday / Asana: PM Skill augments them (doesn't replace). Makes them smarter.

Custom GPTs: Everyone reinvents the wheel. PM Skill is pre-built, shareable, documented.

Blue Ocean Opportunity

No one is building AI agent skills for PMs.

Everyone builds SaaS tools or general automation. PM Skill occupies unique position:

  • Open-source (vs. SaaS)
  • Agent-native (vs. UI-first)
  • PM-specific (vs. general automation)
  • Cross-platform (vs. vendor lock-in)

Competitive Moat: First-mover advantage + open-source network effects.

Risks & Mitigations

Risk 1: Platforms Change Too Fast

Mitigation: Version pinning, minimal dependencies, backwards compatibility.

Risk 2: Low External Adoption

Mitigation: Pre-launch audience building, seed users, quality over hype, aggressive LinkedIn promotion.

Risk 3: Enterprise Privacy Concerns

Mitigation: Local-first design, privacy mode, clear "data never leaves your machine" docs.

Risk 4: Jira API Rate Limits

Mitigation: Rate limit handling, caching, batch operations, user education.

Risk 5: Rajan Gets Busy

Mitigation: Minimal maintenance design, recruit co-maintainers early, archive gracefully if needed.
