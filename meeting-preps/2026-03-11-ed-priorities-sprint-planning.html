Friday COO · Sprint Planning Brief

ED Priorities Consolidation

Built from tonight’s ED Priorities and ED Priorities V2 transcripts. This is the operator’s brief for sprint planning: what Khushbu actually wants, what must move now, what to keep pragmatic, and what needs tight alignment with Greg.

Prepared 11 Mar 2026
Sources: ED Priorities + ED Priorities V2
Audience: Rajan

Executive read: Khushbu is not asking for a grand architecture debate. She is asking for visible control: time-boxed foundation delivery, clearer ownership, faster response loops, an inspectable status surface, and enough structure that she can defend the plan upward without surprises.

What Khushbu actually wants

1. Foundation work must stop floating

  • Repo / workflow split / Jira migration / metadata-type work cannot remain open-ended.
  • She wants a time-boxed first cut, not more broad architecture motion.
  • Jira should be the proving ground, not the end-state for every source on day one.

2. A visible response to the MCP / Workday access scare

  • The Access Control Dashboard is now politically important.
  • Expected shape: who has access, who accessed what, anomaly view.
  • She is pragmatic about scope: version 1 is fine if it is real and demoable.

3. One operational view

  • She wants one place to inspect onboarded sources, live use cases, pipeline, ownership, timelines, and leave impacts.
  • She does not want to chase multiple pages or rely on verbal memory.

4. Stronger execution discipline

  • “On track” is not a sufficient status.
  • She wants earlier checkpoints, earlier correction of bad approach, and more active coaching from leads.

What actually needs to be done now

Define this week’s foundation deliverable properly

  • Repo structure
  • Workflow split pattern
  • Enough schema direction to proceed
  • Jira as the pilot

Build Access Control Dashboard v1

  • Tab 1: access mapping
  • Tab 2: observed access
  • Tab 3: anomaly starter tab with 1–2 real scenarios
  • Plain-English permission explanation where possible

Translate technical-foundation work into visible Jira items

  • Use target dates, not fake hard due dates.
  • Make the work legible in sprint views so Khushbu is not blind.

Make status self-serve

  • Reuse the existing summary + drill-down pattern.
  • Keep the underlying Jira / page current enough that Khushbu can inspect without asking.

Surface leave in the same status context

  • Calendar invites alone are not enough.
  • Leave needs to be visible where project status is visible.

What needs to be pragmatic

Do not overbuild anomaly detection now

  • Prototype credibility matters more than completeness.
  • A tiny but real anomaly view is enough for v1.

Use Jira as the proving ground

  • Prove the pattern on Jira first.
  • Generalize after the implementation pain is real and visible.

Use Greg for structure, quality, and continuity

  • Oversight
  • Architecture guidance
  • CI/CD support / unblock
  • Continuity if Bruno disappears on leave mid-stream

Hold the TP line

  • Finish open tickets by 16 March
  • No new features
  • Then lights-on / operationalization only

What Rajan needs aligned with Greg

1. Exact Friday definition of done for Bruno

  • What absolutely must land this week?
  • What is explicitly deferred?
  • What is the backup if Bruno leaves before it is stable?

2. What Greg is personally taking off Bruno’s plate

  • CI/CD support
  • Review and technical guardrails
  • Continuity if Bruno is away

3. Coaching cadence for weaker execution

  • Earlier approach checks
  • Mid-task checkpoints
  • Do not wait until final delivery to discover misunderstanding

4. Workday review scope

  • What a one-hour high-level review should cover
  • What risk it is meant to catch before end-of-quarter testing

5. Native ABAC feasibility

  • Do a small viability check with Claudio / Aaron.
  • Do not let it derail current delivery.

6. Make technical foundation work visible in Jira

  • Khushbu currently feels blind in sprint views.
  • That visibility gap itself is now part of the problem.

Political read

Khushbu is less worried about technical elegance than about drift. She wants enough structure, visibility, and response speed that she is not exposed upward by vague status language or slow-moving foundation work.

She is effectively asking Rajan and Greg to become the stronger control layer: structure the work, tighten checkpoints, reduce surprise, and make the delivery story legible.

Recommended sprint-planning frame

Hard outcomes this sprint

  • Jira / foundation pilot
  • Access Control Dashboard v1
  • Status / reporting cleanup
  • TP to March 16

Explicit non-goals

  • No full metadata redesign
  • No full ABAC migration now
  • No perfect anomaly engine

Operating discipline

  • Named owners + backup owners
  • Mid-sprint checkpoints
  • No end-of-week “trust me” status
